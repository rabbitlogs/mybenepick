# 혜택PICK!! 글쓰기 가이드 (Claude 프로젝트 지식용)

이 문서는 혜택PICK(mybenepick.com) 블로그 글을 작성하고 사이트를 관리할 때 Claude가 항상 따라야 하는 규칙이다. AI 생성 + 최소 검수 파이프라인이므로, 사람이 매번 톤·형식을 교정하지 않아도 되도록 이 문서의 규칙을 기본값으로 삼는다.

**개정 이력**: 2026-07-14 — 22개 글 일괄 발행 후 세션에서 얻은 인사이트(구 04번 문서)를 이 문서에 병합. 같은 날 청년월세지원 글의 핵심 주장("상시 신청으로 전환")이 사실과 다르다는 게 확인되어 22개 글을 전체 삭제하고, 팩트체크 원칙(9장)을 재발방지 조항으로 강화함. 2026-07-15 — 이미지 소스 우선순위를 "캡처 → 표 → 생성"에서 "표 → 생성 → 캡처"로 변경(4-2). 정부 원본 캡처는 사진·화면 자체가 공공누리 자유이용 대상이 아닌 경우가 많아 실사용으로 이어지는 비율이 낮다고 판단. 같은 날 — 발행 전 검수용 HTML 주석이 frontmatter `---`보다 앞에 위치해 Astro 빌드가 실패하는 사고가 반복 확인되어, 7-1(파일 첫 줄은 반드시 `---`)을 신설하고 10장 체크리스트에 반영. 같은 날 — `==하이라이트==` 개수 상한을 1-2개에서 2-3개로 완화(3장, 10장). 실제 발행 글에서 2개가 자연스러운 기본값으로 쓰이고 있어 관행에 맞춤. `<strong class="key">`는 라벨 전용 용도를 유지하며 완화 대상에서 제외. 같은 날 — 외부링크 URL이 실제로는 부정확하거나(도메인은 맞지만 더 정확한 전용 신청 시스템이 따로 있는 경우) 검증 없이 들어가는 사고가 확인되어, 5-1-1(URL은 넣는 시점에 직접 열어 확인)을 신설하고 10장 체크리스트에 반영. 2026-07-16 — 다른 운영 블로그(Rabbit Logs)의 3단계 강조 위계를 참고해, `<strong class="key">`를 라벨 전용에서 "라벨형+본문 개념어형" 겸용으로 확장(3장, 3-1 신설). 본문 중 그 글의 핵심 개념어 1~2개에 브랜드 컬러 색강조를 쓸 수 있게 됐고, 기존 라벨형 용도(`[[SUMMARY]]` 안 대상/금액/기간)는 그대로 유지. 코드 변경(`markdown.mjs`, CSS)이 함께 필요하며 별도 코드 세션에서 적용 예정. 같은 날 — Rabbit Logs와 표로 재대조하는 과정에서 세 가지 불일치 확인 후 정리: ①개념어형 허용 개수를 "소수/하나/1~2개"로 세 곳에서 다르게 표현하던 걸 "1~2개"로 통일, ②SUMMARY 라벨과 헷갈리는 자리(콜론 앞)를 피하라는 규칙이 체크리스트(3-1)에만 있고 3장 본문 설명에는 빠져 있어 3장 본문에도 명시, ③섹션 번호가 3-3부터 시작해 3-1·3-2가 실종돼 있던 걸 3-1(개념어 색강조)·3-2(특수문자 주의)·3-3(SUMMARY)로 순서대로 재정렬하고 전체 참조 링크 일괄 수정. 같은 날 — 발행 대기 21개 글을 경쟁 사이트와 포맷 비교한 결과를 15장(검토 대기)으로 신설했다가, 그중 SUMMARY 항목 순서(3-3에 원칙 추가)·표 헤더 열 순서(2장 6번에 명문화)·FAQ 개수 실측 인사이트(14-5에 추가) 세 가지는 결론이 나서 본문으로 옮기고 15장에서 제거. "제외 대상" 나열 포맷과 신청 절차 소요 기간 병기, 핵심 수치 배지 세 가지는 아직 결론이 안 나 15장에 검토 대기로 남겨둠. 같은 날 — 남은 세 항목도 결론을 냈다: 소요 기간 병기는 채택해 2장 8번·2-3·10장에 반영, 제외 대상 나열은 전용 문법 신설 없이 2-3 체크리스트에 "3개 이상이면 불릿" 정도로만 가볍게 반영, 핵심 수치 배지는 톤 원칙 충돌로 폐기. 15장은 결론 요약만 남기고 비움. 같은 날 — 발행 대기 글에서 FAQ의 Q와 A 사이에 빈 줄 없이 줄바꿈만 쓰면 렌더링 시 한 줄로 붙어버리는 문제를 실제 코드(`markdown.mjs`, `astro.config.mjs`)와 `markdown-it` 렌더링 재현으로 확인. Q 줄 끝에 백슬래시(`\`) 하드 브레이크를 붙이는 규칙을 2-4로 신설하고 2장 9번·10장에 반영. 21개 발행 대기 글 전체에 일괄 적용하면서 이전에 놓쳤던 위반 사례(표 셀 강조를 `==` 대신 `<strong class="key">`로 쓴 것, 라벨과 헷갈리는 본문 개념어형, `eligibility.employmentStatus`의 `any` 혼용, SUMMARY에서 `대상`이 최우선이 아니었던 것, 제외 업종 4개를 문장으로 풀어쓴 것)도 함께 재점검해 수정. A 답변 둘째 줄 들여쓰기는 CSS+렌더링 로직 변경이 필요해 코드 세션 대기 목록(13-1)으로 넘김. 2026-07-20 — 네이비 리브랜딩 코드 세션(홈/카테고리/글상세/About/find 전면 개편, 목차 sticky 사이드바+스크롤스파이, 3줄 요약 전용 컴포넌트 분리, FAQ 정렬 CSS Grid 전환, 모바일 가로탭 네비 등)에서 반복적으로 나타난 실패 패턴(시안-코드 대조 누락, CSS 폭 추정 오류, flex 텍스트 줄바꿈, 전역 셀렉터 충돌, 스코프 CSS 미적용, 날짜 경계값 버그, 빌드성공과 렌더링검증의 혼동)을 16장(코드/디자인 작업 원칙)으로 신설. 같은 날 — 카테고리 리스트·검색결과 우측 수치 배지(`metric`)와 신청 마감 자동 표시(`deadline`) 필드가 코드에는 반영됐으나 가이드 문서에 누락돼 있던 것을 발견해 6-5·6-6으로 신설하고 frontmatter 스키마·10장 체크리스트에 반영. 같은 날 — `metric` 필드에 "무엇을 어떻게" 채워야 하는지 구체적 프롬프트가 없어 사람이 카테고리 리스트 화면을 보고서야 값이 잘 뽑혔는지 확인할 수 있던 상태였다. 실제 발행된 21개 글의 `metric` 값을 전수 분석해 금액형(상한/주기/확정액)·비율형·상태변화형 세 유형과 우선순위 판단 기준(금액 우선, 단 글의 핵심 주장이 자격·요건이면 상태변화형)을 6-5에 추가. 2026-07-21 — 발행 대기 21개 글의 `deadline`을 정부 1차 출처로 전수 확인하는 과정에서, "특정 마감일 없음"으로 묶였던 글들 상당수가 사실은 성격이 다르다는 게 확인됐다. 법정 급여성 제도처럼 예산·인원 캡 자체가 없어 진짜 상시인 경우와, "선착순 1000명"·"예산 소진 시" 같은 캡이 있어 접수는 열려 있어도 소진되면 조기 마감될 수 있는 경우가 `ongoing` 하나로 뭉쳐 있으면 독자가 "천천히 신청해도 된다"고 오해할 수 있다는 문제 제기가 있었다. 이에 따라 `deadline`에 `"budget-limited"` 값을 신설해 두 상태를 구분하기로 했다(6-6 반영). 실시간 예산 소진율은 대부분 공개되지 않으므로 정확한 마감일을 추정하려 하지 않고, "소진 가능성이 있다"는 상태 자체를 표시하는 용도로만 쓴다. 이 결정은 표시 방식(배지 문구 등) 확정을 위해 코드 세션에서 후속 반영이 필요하다(13-1 참고). 같은 날 — `budget-limited`로 태깅된 글을 실제로 계속 사이트에 노출해도 되는지 논의한 끝에, 실시간 소진 여부 판별은 애초에 불가능하므로 판별을 시도하지 않기로 했다. 대신 이 값이 붙은 글에는 "예산 소진 시 조기 마감될 수 있다"는 안내를 상시·고정으로 노출하는 정적 배너 방식으로 처리하기로 확정했다(13-1에 반영). 이 방식은 개별 글을 사람이 주기적으로 재확인하는 수동 프로세스를 대체한다 — 안내가 이미 늘 떠 있으므로 별도 재확인 없이도 독자에게 최신 상태 확인을 유도할 수 있다는 판단. 같은 날(코드 세션) — 13-1에 대기 중이던 두 항목을 실제로 처리했다. ①FAQ A 답변 hanging indent는 em 단위 폭 추정 대신 `grid-template-columns: max-content 1fr`로 라벨 폭을 자동 계산하는 CSS Grid 방식으로 구현(폰트 렌더링 폭 추정이 계속 어긋나던 문제 해결, 16-2 참고). ②`deadline: "budget-limited"`는 스키마에 리터럴 추가 후 목록 배지("소진 시 마감", 앰버 톤)와 상세 배너를 신설했고, 다른 곳(콘텐츠 세션)에서 이미 4개 글에 이 값을 써둔 상태로 리포를 받았는데 스키마 미반영으로 `astro build`가 실패하는 상태였다 — 코드 세션과 콘텐츠 세션이 분리되어 있을 때 한쪽에서 새 값을 도입하면 다른 쪽 스키마도 같이 갱신해야 빌드가 안 깨진다는 걸 재확인(16-4, 16-6 관련).

## 0. 이 블로그가 무엇인가

- **주제**: 새로 생기거나 바뀐 한국 정부 지원금·복지 제도. 롱테일 SEO 전략 — 개별 정책 딥다이브, 신설 정책, 지역 특화 정책 중심으로 쓰고 대형 아그리게이터 사이트와 겹치는 광범위 키워드는 피한다.
- **목표 독자**: 특정 지원금이 "나한테 해당되는지", "언제 얼마나 어떻게 신청하는지"를 빠르게 확인하러 온 사람. 검색에서 바로 들어와서 필요한 정보만 얻고 나가는 경우가 대부분이다.
- **수익 모델**: 애드센스. 검색 유입과 체류시간이 중요하므로 정확성과 가독성이 곧 성과다.

## 1. 톤 — "담백한 대화체"

- 문장 종결은 **~해요체**를 기본으로 한다. ~습니다체는 쓰지 않는다.
- 과장된 감탄, 물음표 남발, 이모지 도배를 하지 않는다. 느낌표는 사실상 쓰지 않는다.
- "정부가 드디어!", "완전 대박 혜택" 같은 낚시성 표현 금지. 사실을 담백하게 전달하되 친절한 안내자 톤을 유지한다.
- 문장은 짧게 끊는다. 한 문장에 정보를 욱여넣지 않는다.
- 독자를 향해 직접 말하듯 쓴다("~하셨다면", "~인 분들이라면") — 논문체나 보도자료체로 흐르지 않도록 주의.

**좋은 예**: "소득 때문에 신청을 미뤘던 분들이 있다면 눈여겨볼 만한 소식이에요."
**나쁜 예**: "본 제도는 소득 기준을 충족하지 못하여 신청이 어려웠던 국민들에게 희소식이 될 전망입니다!"

### 1-1. 대형 키워드는 제목·description을 손해회피형으로

대형 보편 키워드(에너지바우처, 국민연금 감액 등)는 토스뱅크·정부24·SEO 최적화 블로그가 이미 상위를 채우고 있는 경우가 많다. 주제 선정 단계에서 먼저 그 키워드로 직접 검색해보고, 상위 10개가 전부 대형 은행·정부·SEO블로그면 더 좁은 롱테일 각도로 틀거나 스킵을 고려한다.

이미 대형 키워드로 잡은 글이라면, 최소한 **제목과 description**은 정보 요약형이 아니라 손해회피형·구체적 수치형으로 쓴다.
- 나쁜 예: "에너지바우처 최대 70만원, 12월까지 신청하세요" (정보 요약)
- 좋은 예: "신청 안 하면 사라지는 70만원, 에너지바우처" (손해회피)

title은 6장 기준(30자 내외)을 유지하면서 압축한다. 클릭 유도 요소를 넣으면 글자수가 늘어나기 쉬우므로, 이모지·과장 없이 핵심 손해/숫자만 남기고 나머지는 쳐낸다.

이 원칙을 적용해도 낚시성 표현("완전 대박", "드디어!")은 여전히 금지(위 톤 규칙 그대로 유지). 손해회피와 낚시는 다르다 — 손해회피는 사실에 기반한 긴장감이고, 낚시는 과장이다.

### 1-2. 이모지 사용 범위 — 팁박스에만, 소제목엔 안 씀

- **쓰는 곳**: 3줄 요약 박스 헤더(`📌`), 실무 디테일 팁박스 헤더(`⚠️`) — 딱 이 두 군데만.
- **안 쓰는 곳**: `##` 소제목, 본문 문장, FAQ 질문, 캡션. 전부 텍스트만.
- 토스뱅크류는 소제목마다 이모지를 붙이는 스타일(예: "⚡ 뭐예요?")인데, 이건 혜택PICK의 담백한 톤과 안 맞아서 의도적으로 채택하지 않는다.

## 2. 글 구조 (표준 템플릿)

아래 순서를 기본 골격으로 삼되, 주제에 안 맞는 섹션은 생략 가능하다. 이미지는 표로 담기지 않는 정보가 있을 때만 넣는 선택 요소다(4장 참고). 글을 쓰기 시작하는 시점에 "이미지를 넣을지, 넣는다면 자리가 어디인지"를 먼저 정하고 그 흐름에 맞춰 앞뒤 문장을 쓴다 — 다 쓴 뒤에 빈 자리를 찾아 끼워 넣지 않는다.

1. **도입 (2~3문장)**: 가장 중요한 사실(무엇이 언제 바뀌는지, 숫자)을 첫 문장에 바로 제시. `==하이라이트==`로 핵심 문장 강조.
2. **3줄 요약 박스** (아래 2-1 참고).
3. **`[[TOC]]`** 마커 삽입 (목차 자동 생성 — 소제목이 2개 이상일 때만 실제로 렌더링됨).
4. **배경/이 제도가 왜 있는지** (선택) — 처음 접하는 독자를 위한 맥락. 너무 길게 늘어지지 않게 2~3문단.
5. **제도 정의** — "OO란" 소제목으로 짧고 명확하게 재정의.
6. **핵심 변화 내용** — 표(기존 vs 변경)가 있으면 가장 좋음. 핵심 셀은 `==하이라이트==`. 표 바로 다음에 캡션(2-2 참고). 이미지를 넣기로 했다면 이 섹션이 기본 자리다(4장 참고). 표 헤더는 **기준·구분 열을 첫 열에, 비교 대상을 그 다음 열에** 두는 순서를 지킨다 — 명칭은 글마다 다를 수 있다(`구분`/`유형`/`가구원 수` 등 첫 열, `기존`/`변경`/`청년미래적금`/`요건심사형` 등 다음 열). 첫 열과 다음 열의 위치가 뒤바뀌면 표를 훑는 순서가 글마다 달라져 일관성이 깨진다.
7. **대상/금액/조건** — `<strong class="key">라벨</strong>: 설명` 형식으로 항목화하되, **`[[SUMMARY]]...[[/SUMMARY]]`로 감싸서 "한눈에 보기" 요약 박스로 렌더링한다** (3-3 참고). 검색으로 바로 들어와 조건만 빠르게 확인하고 나가는 독자가 많은 사이트 특성상, 이 섹션은 본문 서술형 문장보다 요약 박스 우선을 기본값으로 한다.
8. **신청 방법** — 번호 목록으로 절차 나열. 각 단계에 확인된 소요 기간(심사 기간, 처리 기간 등)이 있으면 그 단계 문장 안에 괄호로 바로 병기한다(예: "관할 고용센터에서 심사 후 결과를 통보해요(약 1개월 이내)") — 절차와 기간 정보를 따로 떼어놓지 않는다. 이 섹션에 공식 신청 페이지 등 외부링크를 자연스럽게 포함 (5-1 참고). 실무 디테일이 확인되면 `> ⚠️` 팁박스로 보강 (2-3 참고).
9. **자주 묻는 질문 (FAQ)** — `**Q. 질문**\` / `A. 답변` 형식, 2~4개. **Q 줄 끝에 반드시 백슬래시(`\`)를 붙인다** (아래 2-4 참고) — 붙이지 않으면 Q와 A 사이에 줄바꿈이 렌더링되지 않고 한 줄로 붙어 보인다.
10. **더 읽어보기** (관련 글이 있을 때만) — 내부링크 2~3개 (5-2 참고).
11. **마무리 안내 문구** — 이탤릭체로 "이 글은 OOOO년 O월 기준 정보이며, 실제 신청 전 공식 채널에서 최신 정보를 확인하라"는 면책 문장. 항상 포함.

### 2-1. 3줄 요약 박스

도입 문단 바로 다음, `[[TOC]]` 마커 앞에 삽입한다.

```markdown
> 📌 3줄 요약
>
> 1. (핵심 지원 내용 — 숫자 포함)
> 2. (핵심 기간/조건 — 날짜 포함)
> 3. (실무적으로 가장 놓치기 쉬운 포인트)
```

- 이모지는 `📌` 헤더용 1개만. 리스트 항목에는 이모지를 쓰지 않는다.
- 3줄은 도입부 내용을 반복하는 게 아니라, 스크롤 없이 핵심을 파악할 수 있도록 압축한 요약이어야 한다.
- 표나 본문과 숫자가 100% 일치해야 한다(9장 팩트체크 원칙 동일 적용).

### 2-2. 표·이미지 캡션 번호 매기기

- **표 캡션**: 표 바로 다음 줄에 `*표 1. [출처] 기준*` 형식. 한 글에 표가 여러 개면 순서대로 표 1, 표 2...
- **이미지 캡션**: 캡션 문장 맨 앞에 `그림 1.`을 붙인다. `*그림 1. [캡션 내용]*` 형식. 한 글에 이미지가 여러 개면 순서대로 그림 1, 그림 2...
- 표와 그림은 서로 다른 번호 체계로 각각 1부터 센다(통합해서 세지 않음).
- 목적: 화면에 보이는 캡션만으로 "몇 번째 표/그림인지" 참조해서, 여러 md 파일을 오갈 때 위치를 빠르게 찾기 위함.

### 2-3. 실무 디테일 팁박스 — "정확성"과 "실무 유용성"은 별개

정확한 숫자·톤·구조를 다 지켜도, "신청할 때 실제로 부딪히는 디테일"이 빠지면 경쟁 글 대비 밀린다. 신청 방법/조건 섹션을 쓸 때마다 아래를 확인한다.

- [ ] 신청·정산 시스템이 일시적으로 멈추는 기간이 있는가
- [ ] 심사·처리에 걸리는 실제 소요 기간(영업일/개월 단위)이 있는가 — 단순 "며칠 걸린다"는 사실 자체는 2장 8번 규칙대로 해당 절차 단계 안에 괄호로 병기하고, 여기 팁박스는 "그 기간 동안 주의할 점"(예: 지연 시 대처법, 중간 상태 확인 방법)처럼 부가 설명이 있을 때 쓴다.
- [ ] 흔히 놓쳐서 탈락하거나 반려되는 사유가 있는가 (서류, 계좌 방식, 기한 등) — 제외 대상·업종이 3개 이상으로 많으면 문장으로 풀지 말고 팁박스 안에서 불릿으로 나열한다.
- [ ] 부결·탈락 시 재신청이 가능한가, 가능하다면 조건이나 대기 기간은
- [ ] 신청 이후에도 지켜야 할 의무(변동사항 신고, 근속 유지 등)가 있고, 안 지키면 어떤 불이익이 있는가
- [ ] 이의신청이나 소급 적용처럼 "놓쳐도 구제 가능한 경로"가 있는가

확인되면 `> ⚠️` 팁박스로 넣는다. 이모지는 `⚠️` 하나만 박스 맨 앞에, 본문 문장 안에는 넣지 않는다.

### 2-4. FAQ Q-A 줄바꿈 — Q 끝에 백슬래시(`\`) 필수

이 사이트는 `remark-breaks` 같은 줄바꿈 보정 플러그인을 쓰지 않고 표준 마크다운(CommonMark) 그대로 렌더링한다(코드로 직접 확인함 — `astro.config.mjs`의 `remarkPlugins`는 `remarkSummaryBox`/`remarkHighlight`/`remarkInlineToc` 세 개뿐이고 FAQ 전용 처리 로직은 없다).

**문제**: `**Q. 질문**`과 `A. 답변`을 빈 줄 없이 그냥 줄바꿈만 해서 쓰면(즉 아래처럼),

```markdown
**Q. 신청 기간은 언제까지인가요?**
A. 2026년 12월까지예요.
```

CommonMark 표준상 두 줄이 **하나의 `<p>` 태그로 합쳐지고** 그 사이 줄바꿈은 HTML에서 무시된다. 실제로 렌더링해보면 "Q. 신청 기간은 언제까지인가요? A. 2026년 12월까지예요."처럼 한 줄로 붙어 보인다(2026-07-16 `markdown-it`으로 재현 확인).

**해결**: Q 줄 끝에 백슬래시(`\`)를 붙여 마크다운 하드 브레이크를 만든다.

```markdown
**Q. 신청 기간은 언제까지인가요?**\
A. 2026년 12월까지예요.
```

이러면 Q와 A는 여전히 하나의 `<p>` 안에 있지만 `\`가 `<br />`로 변환돼서 줄만 바뀐다 — Q-A는 붙어 보이고, 다음 FAQ 쌍(새 `**Q.**`)은 별도 `<p>`라서 `.prose p { margin: 1.15rem 0; }` 여백이 그대로 유지돼 쌍 사이는 넓게 벌어진다.

- **줄 끝 공백 2칸(trailing space)이 아니라 `\`를 쓴다.** 둘 다 CommonMark 하드 브레이크로 동작하지만, 공백은 파일에서 눈에 안 보여서 AI 생성 파이프라인에서 다음에 파일을 다시 저장·편집할 때 실수로 잘려나가기 쉽다. `\`는 파일을 열어봐도 명확히 보여서 안전하다.
- **A 답변이 길어서 화면에서 자동으로 두 번째 줄로 넘어갈 때, 그 줄이 들여써지지 않는 건 별개 문제다.** 이건 마크다운 텍스트로 해결이 안 되고 CSS(hanging indent) + 렌더링 로직(A 텍스트를 `<span>`으로 감싸는 remark 플러그인) 양쪽이 필요한 코드 작업이라, 코드 세션에서 별도로 진행한다(13장 참고).

## 3. 마크다운 문법 규칙

이 사이트 전용 커스텀 문법이 있다. 일반 마크다운과 혼동하지 말 것.

강조는 3단계 위계로 쓴다. 색을 많이 쓰면 독자가 핵심을 못 고르므로 절제한다.

| 문법 | 위계 | 용도 | 예시 |
|---|---|---|---|
| `**텍스트**` | 1단계 — 일반 강조(검정) | 핵심 단어·문구, 가장 흔하게. FAQ의 Q에도 사용 | `**Q. 신청 기간은 언제까지인가요?**` |
| `<strong class="key">텍스트</strong>` | 2단계 — 브랜드 컬러 색강조 | ①`[[SUMMARY]]` 안에서는 "대상/금액/기간" 라벨, ②본문에서는 그 글의 핵심 개념어(글당 1~2개) | `<strong class="key">대상</strong>: 한부모가구` / `이 제도의 핵심은 <strong class="key">소득기준 폐지</strong>예요` |
| `==텍스트==` | 3단계 — 배경색 하이라이트 | 그 글에서 가장 핵심인 문장에만 | `==2026년 10월 29일부터 소득기준이 없어져요.==` |
| `[[TOC]]` | — | 목차 자동 생성 마커. 소제목 2개 이상일 때만 렌더링됨 | 도입부 끝, 첫 `##` 앞에 배치 |
| `[[SUMMARY]]...[[/SUMMARY]]` | — | "한눈에 보기" 요약 박스. 대상/금액/기간 등을 카드 형태로 렌더링 | 3-3 참고 |
| `## 소제목` | — | h2, 목차에 잡힘 | |
| `### 소소제목` | — | h3, 목차에 하위 항목으로 들어감 | |
| `\| 표 \|` | — | 기존 vs 변경 비교에 적극 활용. 헤더 열 순서는 2장 6번 참고 | |

**2단계 색강조(`<strong class="key">`)는 두 가지 맥락에서 쓰인다 — 용도가 다르니 섞어 쓰지 않는다.**

- **라벨형**(`[[SUMMARY]]` 안): "대상", "금액", "기간" 같은 항목 라벨 전용. 3-3 참고.
- **개념어형**(본문 중): 그 글에서 가장 중요한 핵심 개념어에만 색을 입혀 눈에 띄게 한다. "이게 이 글의 결론"이라기보다 "이 단어가 이 글의 주인공"에 가까운 자리 — 예를 들어 제도 이름 자체, 핵심 조건의 이름(소득기준, 사후지급금 등)에 쓴다. 문장 전체가 아니라 짧은 단어·구 단위로만 씌운다. 글당 **1~2개**로 제한한다(3-1 참고).
- 라벨형과 개념어형을 같은 문단에서 동시에 남발하지 않는다. 특히 문장 안에 콜론(`:`)이 오는 자리에 개념어형을 쓰면 라벨형처럼 보여 혼동을 준다 — 개념어형은 콜론 앞이 아니라 문장 서술 안에 자연스럽게 녹여 쓴다.

**하이라이트(`==`) 사용 원칙**: 남발하지 않는다. 글 하나에 `==하이라이트==` **2~3개**(도입부 핵심 사실 1개 + 본문 실무 핵심 1개 + 결론 문장 1개 정도가 기본). 모든 문장을 강조하면 아무것도 강조되지 않은 것과 같다. 상한선을 넉넉하게 잡으면 실제로는 그 상한까지 채워지는 경향이 있으므로, 필요한 개수보다 여유 있게 상한을 잡지 않는다 — 2~3개도 이미 여유를 준 상한이니 3개를 채우는 게 기본값이 되지 않도록 주의한다. 발행 전 체크리스트(10장)에서 `==` 개수는 글을 다 쓴 뒤 실제로 세어보는 별도 단계로 취급한다 — 쓰면서 감으로 판단하지 않는다.

→ 강조 3단계(굵게/색강조/하이라이트)는 서로 겹쳐 쓰지 않는다. 한 구간에 `**`와 `<strong class="key">`를 동시에 씌우거나, `==` 안에 `<strong class="key">`를 중첩하지 않는다 — 위계가 무너진다.

### 3-1. 본문 개념어 색강조 — 라벨형과 헷갈리지 않기

본문 개념어형 `<strong class="key">`는 다음 기준으로 판단한다.

- [ ] 이 단어가 그 글에서 실제로 가장 중요한 명사(제도명·조건명·숫자 기준 등)인가
- [ ] 문장 전체가 아니라 짧은 단어·구 단위인가 (긴 절 전체를 씌우면 하이라이트와 역할이 겹친다)
- [ ] 글당 1~2개를 넘지 않는가 — 넘으면 정말 핵심인 것만 남기고 나머지는 일반 굵게(`**`)로 낮춘다
- [ ] `[[SUMMARY]]` 라벨과 헷갈릴 자리(콜론 앞)에 쓰고 있지 않은가 — 라벨형과 개념어형이 한눈에 구분되지 않으면 개념어형 사용을 보류한다

**좋은 예**: "이번 개정의 핵심은 <strong class="key">소득기준 폐지</strong>예요. 기존에는 부모 소득이 중위소득 150%를 넘으면 대상에서 제외됐지만, 이제는 소득과 무관하게 신청할 수 있어요."
**나쁜 예**: "이번 개정의 핵심은 <strong class="key">소득기준이 폐지되어 이제는 누구나 신청할 수 있게 됐다는 점</strong>이에요." (문장 전체를 씌워 하이라이트와 역할이 겹침)

### 3-2. 마크다운 특수문자 주의 — 범위·기호 표기

**물결표(`~`) 금지 — 취소선으로 오작동한다.** 실제로 이 사이트에서 `3~6개월`을 쓰면 `3<del>6개월이에요. 금액은 300</del>500만원`처럼 취소선(`<del>`) 태그가 씌워지면서 문장이 통째로 깨지는 걸 빌드로 확인했다. 정부 지원금 글은 "3~6개월", "월 20~30만원" 같은 범위 표현이 잦으므로 특히 주의한다.
- **금지**: `3~6개월`, `300~500만 원` (물결표)
- **권장**: `3-6개월` (하이픈) 또는 `3개월에서 6개월`처럼 풀어 쓴다.

**굵게(`**`) 닫기 주의 — 괄호+조사 조합도 깨진다.** 굵게를 닫는 `**` 바로 앞이 닫는 괄호 `)`이고 뒤에 한글 조사가 바로 붙으면, `**`가 그대로 텍스트로 노출되고 강조가 적용되지 않는 걸 빌드로 확인했다 (예: `**포인트(CEU)**를` → 화면에 `**포인트(CEU)**를`이라는 글자 그대로 보임). 일반적인 "굵게+조사"(`**전사적 자원관리**예요`)는 정상 작동하니, **괄호로 끝나는 경우만** 주의한다.
- **해결**: 강조 범위를 괄호 앞까지만 잡는다. `**포인트**(CEU)를`, `**지원금**(바우처)은`처럼 괄호는 강조 밖에 둔다.
- 같은 원리로 `<strong class="key">` 태그도 닫는 태그 바로 앞이 괄호이고 뒤에 조사가 붙으면 동일하게 깨질 가능성이 있다. 본문 개념어형에 괄호가 포함된 용어를 쓸 때는 굵게와 같은 방식으로 괄호를 강조 밖에 둔다.

- 그 밖에 마크다운 특수문자(`*`, `_`, `` ` ``, `#`, `>`)를 일반 텍스트로 쓸 때도 의도치 않은 서식이 적용되지 않는지 미리보기로 확인한다.

### 3-3. `[[SUMMARY]]...[[/SUMMARY]]` — "한눈에 보기" 요약 박스

검색으로 바로 들어와서 "나한테 해당되는지"만 빠르게 확인하고 나가는 독자가 많은 사이트 특성을 반영해 만든 전용 문법이다. 2장 템플릿의 대상/금액/조건 섹션에서 기본으로 사용한다.

**문법**

```markdown
[[SUMMARY]]
<strong class="key">대상</strong>: 한부모가구
<strong class="key">금액</strong>: 월 20만원
<strong class="key">기간</strong>: 2026년 10월 29일부터 신청 가능
[[/SUMMARY]]
```

- `[[SUMMARY]]`와 `[[/SUMMARY]]` 사이에 **빈 줄을 넣지 않는다.** 빈 줄이 들어가면 마크다운 문단이 나뉘어서 파싱에 실패하고, 마커가 화면에 그대로 노출된다. 시작 마커·항목들·끝 마커를 줄바꿈만으로 붙여 쓴다.
- 각 항목은 반드시 `<strong class="key">라벨</strong>: 설명` 형식이어야 한다.
- 항목 개수는 보통 3~4개(대상/금액/기간/신청처 중 그 글에 해당하는 것)가 적당하다. 너무 많으면(6개 이상) 오히려 "한눈에" 보기 어려워지니, 핵심만 추리고 나머지는 본문 서술로 풀어낸다.
- **항목 순서**: 라벨 자체가 글마다 다른 개념(대상/금액/기간뿐 아니라 보험료율, 자녀 요건, 감액 기준 등)이라 모든 글에 통하는 고정 순서를 강제하지 않는다. 대신 아래 두 가지 원칙으로 순서를 정한다.
  - 숫자·날짜가 담긴 라벨(금액, 한도, 기간, 시행일 등)을 절차성 라벨(신청처, 문의처 등)보다 앞에 둔다 — 독자가 가장 먼저 확인하고 싶은 게 "얼마를, 언제까지"이기 때문이다.
  - "대상"이 있는 글은 대상을 가장 먼저 둔다 — "나한테 해당되는지"부터 확인하는 사이트 특성(0장) 때문이다.
- **형식이 어긋난 항목은 조용히 사라지지 않고 원본 마커(`[[SUMMARY]]` 텍스트 등)가 그대로 화면에 노출된다** — 발행 전 미리보기에서 요약 박스가 카드 형태로 안 보이고 대괄호 텍스트가 그대로 보이면 항목 형식(특히 `<strong class="key">...</strong>` 태그, 콜론)을 다시 확인한다.
- 한 글에 요약 박스는 1개만 쓰는 걸 기본으로 한다. 여러 제도를 종합하는 `deep-guide` 글처럼 제도별로 조건이 확연히 나뉘는 경우에만 예외적으로 2개 이상 검토한다.

## 4. 이미지 — 넣기로 했다면 alt 텍스트와 캡션은 항상 함께

### 4-0. 이미지는 선택 요소다 — 표로 안 되는 정보가 있을 때만

이미지는 글마다 채워야 하는 필수 요소가 아니라, **표로 담기지 않는 정보가 있을 때만 넣는 선택 요소**다.

**판단 기준**: 표나 우리 스타일 생성 라인아트로 재현 가능한 정보인가?
- **재현 가능**(숫자 비교, 절차 흐름, 개념 설명) → markdown 표 또는 생성 라인아트를 우선 사용. 브랜드 톤 일관성과 로딩 속도, 접근성(스크린리더) 면에서 유리하다.
- **재현 불가능**(실제 신청 화면 UI, 정부가 만든 복잡한 원본 산식·인포그래픽, 지도, 원문 조항) → 캡처를 검토하되, 저작권 확인이 까다로운 경우가 많으니 생성 라인아트로 대체 가능한지 먼저 본다(순서는 4-2 참고).

표로 이미 정보가 충분히 전달된다면 이미지를 억지로 추가하지 않는다. "이미지가 없어서 허전해 보이니 하나 넣자"는 판단은 지양한다 — 이미지 하나를 프롬프트→생성→검토→재수정하는 루프는 비용이 크고, 급하게 여러 장을 만들면 장별로 스타일(선 굵기, 음영 유무, 디테일 수준)이 미묘하게 달라져 사이트 전체의 통일감이 떨어지기 쉽다. 체감 분량이 부족하다고 느껴질 때는 이미지보다 2-1(3줄 요약)과 2-3(실무 디테일 팁박스)로 보강하는 걸 먼저 검토한다.

트래픽이 확인된 뒤(예: 검색 유입이 꾸준한 글) 그 글에만 선별적으로 이미지를 추가 투자하는 방식이 전체 글에 균등 투자하는 것보다 효율적이다.

### 4-1. alt와 캡션은 역할이 다르다 — 이미지를 쓴다면 둘 다 넣는다

이 사이트는 이미지를 클릭하면 확대되는 라이트박스 기능이 있고, **alt 텍스트가 그 라이트박스의 설명문으로도 그대로 쓰인다.** alt를 비우면 확대했을 때도 아무 설명이 안 뜨니, alt는 절대 생략하지 않는다.

| 요소 | 문법 | 역할 | 화면에 항상 보이는가 |
|---|---|---|---|
| **alt 텍스트** | `![alt 텍스트](경로)` | 스크린리더용 설명 + 라이트박스 확대 시 하단 설명문 | 평소엔 안 보이고, 이미지 클릭해서 확대했을 때만 보임 |
| **캡션** | 이미지 바로 다음 줄에 `*그림 N. 이탤릭 문단*` | 화면에 항상 보이는 회색 작은 글씨 설명/출처 표기 | 항상 보임 (본문 안에 인라인으로) |

```markdown
![양육비 선지급제 신청 절차를 보여주는 인포그래픽](/images/childcare-process.png)

*그림 1. 양육비이행관리원 공식 자료 기준 신청 절차*
```

**⚠️ 이미지와 캡션 사이에 반드시 빈 줄을 넣는다.** 빈 줄 없이 붙여 쓰면 마크다운이 이미지와 캡션을 한 문단으로 합쳐버려서, 캡션이 회색 작은 글씨로 스타일링되지 않고 이미지 밑에 그냥 이어 붙은 것처럼 보인다. 반드시 이미지 줄 → 빈 줄 → 캡션 줄 순서로 띄어 쓴다.

- alt 텍스트는 "이 이미지가 뭘 보여주는지"를 설명하는 문장으로 쓴다. "이미지1", "그림" 같은 무의미한 텍스트 금지.
- 캡션은 짧게 — 부가 설명이나 출처 한 줄. 캡션은 alt와 내용이 겹쳐도 상관없지만, 가능하면 캡션은 "무엇인지", alt는 "왜 실었는지/무슨 정보가 담겼는지"로 살짝 다르게 쓰는 게 정보량이 많아 좋다.

### 4-2. 이미지 소스 — 표 → 생성 → 캡처 순으로 검토

이 블로그는 표, AI 생성 이미지, 캡처(정부 사이트·공고문 스크린샷) 세 가지를 쓴다. 넣기로 한 이미지라면 아래 순서로 판단한다.

1. **표로 재현 가능한가** → 표로 대체. 대부분의 정부 지원금 정보(대상/금액/기간/절차)는 표로 충분히 전달된다.
2. **표로 안 되면 생성 라인아트로 가능한가** → 생성 라인아트 사용. 저작권 이슈 자체가 없고 톤도 일관되게 유지된다.
3. **그래도 캡처가 꼭 필요하면(실제 신청 화면 UI, 복잡한 원본 인포그래픽, 지도 등 재현 불가능한 경우만)** → 캡처 검토. 단, 아래 이유로 캡처는 최후 수단으로만 쓴다.

**캡처를 최후 수단으로 두는 이유**: 정부 사이트는 대부분 "텍스트는 공공누리로 자유이용 가능, 사진·이미지·화면 자체는 별도 저작권자 확인 필요"인 경우가 많다. 즉 정작 캡처하고 싶은 화면(신청 화면, 인포그래픽 등)은 공공누리 제1유형 확인이 안 되는 경우가 대부분이고, 자유이용이 확인되는 건 텍스트뿐인 경우가 많아 캡처를 시도해도 실제로 쓸 수 있는 결과로 이어지는 비율이 낮다. 따라서 캡처를 전제로 자료조사 시간을 쓰기보다, 표/생성으로 먼저 해결되는지부터 본다.

**캡처를 쓸 때** (위 1, 2번으로 해결이 안 될 때만)
- 정부 부처·지자체·공공기관 공식 페이지, 공고문, 신청 화면 등 출처가 명확한 캡처만 사용한다.
- alt 텍스트에 출처가 드러나게 쓴다: `![복지로 홈페이지의 양육비 선지급제 신청 화면](/images/...)`
- 캡션에 출처 문구를 명시한다: `*그림 1. 출처: 복지로(bokjiro.go.kr)*`
- 개인정보(신청자 이름, 주민번호 등 예시 화면에 노출된 정보)가 있으면 모자이크 처리 후 사용한다.
- 그래도 캡처를 검토한다면 정책브리핑(korea.kr) 카드뉴스/정책뉴스를 먼저 확인한다. 다만 여기도 대부분 "텍스트만 공공누리 자유이용, 사진·이미지는 별도 저작권자 확인 필요"라는 제약이 있다. **공공누리 제1유형(출처표시 조건 상업적 이용 가능)이 명시적으로 확인된 경우만 안전하게 캡처**하고, 표시가 불명확하면 곧바로 표/생성으로 되돌아간다 — 캡처 후보를 더 찾아보는 데 시간을 쓰지 않는다.

**자료 조사 중 캡처 후보를 찾으면 — 보고만 하고, 이용 가능 여부는 사람이 최종 확인한다**

글감을 웹 검색으로 조사하는 과정에서 캡처하면 좋을 화면을 발견하면, 다음 세 가지를 함께 보고한다.

1. **어느 화면인지** — 페이지 내 정확한 위치(예: "OO부 보도자료 페이지 중 신청절차 안내 표")
2. **링크** — 그 페이지의 정확한 URL
3. **이용 가능성에 대해 확인한 사실** — 공공누리(KOGL) 마크 표시 여부와 유형. 상업적 이용(애드센스 수익화)을 감안해 유형을 반드시 확인해서 전달한다.

**Claude는 "공개해도 된다/안 된다"를 확정적으로 판단하지 않는다.** 페이지에서 실제로 확인한 사실만 전달하고, 마크가 없거나 이용조건이 불분명한 페이지는 "이용조건이 명시돼 있지 않음 — 직접 확인 필요"라고 있는 그대로 보고한다. 최종적으로 캡처해서 쓸지 말지는 사람이 그 페이지를 직접 열어보고 판단한다. 특히:
- 마크 없이 저작권 문구만 있는 페이지(예: "ⓒ OOO부, 무단전재 금지")는 자유이용 대상이 아닐 가능성이 높다는 점을 함께 알린다.
- 신청 화면처럼 개인정보 입력 폼이 포함된 캡처는 공공누리 유형과 별개로, 모자이크 규칙이 항상 우선 적용된다.

**보고 형식 예시**:
> 이 글에 넣으면 좋을 캡처 후보를 찾았어요.
> - **화면**: 복지로 홈페이지의 양육비 선지급제 신청 절차 안내 화면
> - **링크**: https://www.bokjiro.go.kr/ssis-tbu/...
> - **이용조건**: 페이지 하단에 공공누리 제1유형(출처표시) 마크가 표시되어 있어요 — 출처만 밝히면 상업적 이용도 가능한 조건이에요. (다만 실제 캡처 전에 페이지에서 직접 한 번 더 확인해주세요.)

**생성 이미지를 쓸 때 — 프롬프트 규칙**

브랜드 마스코트(다람쥐, 안경 쓴 캐릭터)와 톤을 맞추기 위해 아래 스타일을 프롬프트에 기본으로 포함한다:

```
스타일: 미니멀 라인아트, 얇고 균일한 선 굵기, 손그림 느낌의 살짝 불완전한 곡선
색상: 배경은 크림색(#faf8f3), 선 색상은 브라운 계열(#8a5a3f 또는 #6b5133) 단색만 사용 — 다색 배색 금지
분위기: 친근하고 차분함, 과장되거나 화려한 표현 지양
구도: 여백을 좁게, 화면을 꽉 채우는 구도로. 중앙 또는 우측 정렬, 텍스트가 들어갈 자리를 비워두지 않아도 됨(텍스트는 코드가 별도로 얹음)
금지 사항: 실제 인물 얼굴, 브랜드 로고, 정당·정치인 관련 이미지, 사실적(포토리얼) 렌더링
```

용도별로 다음 내용을 덧붙인다:

- **개념 설명용 일러스트** (예: "제도 구조를 도식으로") → `위 스타일로, [설명하려는 개념]을 표현하는 심플한 다이어그램. 화살표나 아이콘으로 흐름을 명시하고, 텍스트 라벨 없이.` 여러 아이콘을 배치한다면 화살표·연결선으로 하나의 흐름("A에서 B로, B에서 C로")을 이루게 한다 — 나란히 배치만 하면 클립아트 나열처럼 보인다.
- **인포그래픽/비교표 이미지** → 표로 표현 가능한 내용은 이미지 대신 마크다운 표를 쓴다.
- **캐릭터 등장 이미지** (다람쥐 마스코트 활용) → `위 스타일로, 안경 쓴 다람쥐 캐릭터가 [상황]. 기존 마스코트와 같은 얼굴 특징(둥근 안경, 짧은 주둥이) 유지.` — 마스코트 얼굴 특징이 매번 달라지지 않도록 `public/images/squirrel-original.png`를 레퍼런스로 함께 제공한다.

### 4-2-2. 이미지 생성 시 스타일 일관성 체크리스트

여러 이미지를 한 번에 만들다 보면 장마다 스타일이 미묘하게 갈리는 경향이 있다. 생성 후 아래를 확인한다. 이 체크리스트를 프롬프트 자체에 미리 반영하면(예: "화살표로 연결", "여백은 좁게") 재수정 횟수를 줄일 수 있다.

- [ ] **음영(그림자·하이라이트) 유무가 통일됐는가** — 순수 아웃라인(선만)인지, 음영이 살짝 들어간 일러스트인지 섞이면 안 된다. 사이트 전체는 순수 아웃라인 라인아트가 기본값이다.
- [ ] **디테일 밀도가 비슷한가** — 어떤 이미지는 디테일까지 그려져 있고 어떤 이미지는 단순 실루엣이면, 같은 사이트 안에서 튄다.
- [ ] **텍스트 라벨이 실수로 들어가지 않았는가** — "텍스트 없이"를 프롬프트에 넣어도 소품에 문자가 딸려 나오는 경우가 있다. 완성본을 확대해서 라벨성 텍스트가 있는지 확인한다.
- [ ] **여러 아이콘을 한 이미지에 배치했다면, 화살표나 연결선으로 하나의 흐름을 이루는가**
- [ ] **여백이 과한가** — 화면의 절반 이상이 빈 배경이면 허전해 보인다.

### 4-3. 이미지 사이즈 규격

이 사이트의 본문 컨테이너 폭은 720px다(`.prose img { max-width: 100% }`). 아래 사이즈는 **화면에 보이는 크기가 아니라 저장하는 원본 해상도** 기준이다 — 720px보다 넉넉하게 원본을 만들어두면 화면에서는 자동으로 축소되지만, 레티나(고해상도) 디스플레이에서도 흐려 보이지 않는다.

- **기본 사이즈(고정)**: 가로 **1200 × 세로 675 (16:9)**로 통일한다.
- 포맷: JPG, 품질 80% 내외, 파일 용량 200KB 이하로 압축 권장. 캡처/스크린샷은 PNG도 무방하나 용량이 커지면 JPG 변환을 검토한다.
- 그보다 크게 캡처해도 본문 폭에 맞춰 자동으로 줄어드니 문제없다(오히려 더 선명). 작게 캡처한 원본은 억지로 늘리지 말고 원본 그대로 쓴다 — 확대하면 화질이 깨진다.
- 글자·선이 많은 스크린샷(신청 화면, 공고문 등)은 특히 1200px 이상으로 캡처해야 확대(라이트박스)했을 때도 텍스트가 읽힌다.
- 16:9 비율이 어색한 이미지(세로로 긴 절차도, 좁은 표 캡처 등)는 억지로 16:9에 맞추지 않는다 — 가로폭 1200px만 고정하고 세로는 내용에 맞게 유동적으로 둔다. 대신 지나치게 세로로 길면(2:1 이상) 정보를 나눠서 2개 이미지로 분리하는 걸 우선 검토한다.
- 다람쥐 마스코트 캐릭터 이미지처럼 정사각형에 가까운 구도가 자연스러운 경우는 16:9를 고집하지 않아도 된다.

### 4-4. 이미지 배치 규칙

- **파일명 규칙**: `글슬러그-순번.jpg` 형식. 소문자 영문 + 하이픈만 쓰고, 한글·공백·특수문자는 금지한다. 예) `childcare-advance-payment-income-limit-abolished-01.jpg`.
- 파일 경로: `public/images/` 아래 저장.
- 본문에는 `/images/파일명.jpg`로 참조 (경로 앞에 `/images/` 필수, `public/` 접두어는 쓰지 않음).
- **위치는 초안 작성 시점에 정한다**: 이미지를 어디에 넣을지는 글을 다 쓰고 나서 빈 곳을 찾아 끼워 넣는 게 아니라, 2장 템플릿의 "핵심 변화 내용" 섹션을 쓰는 시점에 "여기에 어떤 이미지가 들어간다"를 먼저 정하고 그 흐름에 맞춰 앞뒤 문장을 쓴다.
- 글 대표 이미지(소셜 공유용)가 필요하면 frontmatter의 `thumbnail` 필드에 경로를 지정한다. 지정하지 않으면 사이트 기본 OG 이미지로 대체된다.

## 5. 링크 — 모든 글에 외부링크 1개 이상 + 내부링크 섹션 필수

이 사이트는 링크 처리 방식이 특이하게 설계되어 있다. **본문 안의 모든 링크는 기본적으로 새 탭으로 열리도록 자동 처리된다** — 즉 본문 중간에 넣는 링크는 원칙적으로 외부 공식 사이트 링크를 전제로 한다. 내부 링크(다른 글 링크)는 반드시 아래 5-2의 "더 읽어보기" 섹션에만 넣는다.

### 5-1. 외부링크 — 모든 글에 최소 1개 이상 필수

- 본문 서술 중 제도의 근거가 되는 공식 출처를 직접 링크로 건다. 예: "양육비이행관리원 홈페이지에서 신청할 수 있어요" → `[양육비이행관리원 홈페이지](https://www.childsupport.or.kr)`처럼 실제 링크로.
- 최소 1개, 가능하면 2~3개(신청 방법 섹션의 공식 사이트, 자격 조회 시스템, 근거 보도자료 등)를 자연스럽게 본문에 녹여 넣는다.
- 우선순위: 정부 부처·공공기관 공식 사이트 > 관련 법령/고시 원문 > 신뢰할 수 있는 언론 보도. 블로그·카페·개인 SNS는 링크하지 않는다.
- 링크 텍스트는 URL을 그대로 노출하지 말고 "OO 홈페이지", "OO 신청 페이지"처럼 의미가 드러나게 쓴다.
- 이 링크들은 코드가 자동으로 새 탭(`target="_blank"`)으로 열리게 처리하니, 마크다운에서는 그냥 일반 링크(`[텍스트](url)`)로만 쓰면 된다 — 별도 표시 불필요.

### 5-1-1. 외부링크 URL은 넣는 시점에 직접 열어서 확인한다

URL을 기억이나 추정으로 만들어 넣지 않는다. 정부·공공기관 사이트는 조직 개편이나 리뉴얼로 URL이 자주 바뀌고, 특히 게시글·보도자료 URL은 숫자 ID 기반이라 추정이 사실상 불가능하다. "그럴듯한 URL"과 "실제로 존재하는 URL"은 다르다.

- 본문에 외부링크를 넣을 때마다 그 URL을 실제로 열어(web_fetch 등으로) 페이지가 정상적으로 뜨는지, 앵커 텍스트가 가리키는 내용과 실제 페이지 내용이 일치하는지 그 자리에서 확인한다. 발행 직전에 몰아서 검증하지 않는다 — 몰아서 하면 검증을 건너뛰기 쉽고, 무엇을 기준으로 그 URL을 골랐는지도 잊어버리기 쉽다.
- **도메인이 실존해도 "가장 정확한 페이지"가 아닐 수 있다.** 예: 특정 제도의 신청은 소관 부처 메인 사이트가 아니라 그 제도 전용 별도 시스템(예: 전세사기피해자 지원관리시스템처럼 부처 도메인과 다른 별도 도메인)에서 이뤄지는 경우가 있다. 부처 메인 URL이 에러 없이 열린다는 것만으로 충분하다고 판단하지 말고, 검색으로 "그 제도의 실제 신청처가 이 URL이 맞는지"까지 교차 확인한다.
- 접근이 막힌 사이트(robots.txt 등으로 직접 열람이 안 되는 정부 사이트)는 검색 스니펫이나 다른 공식 문서(정책브리핑, 관련 부처 보도자료 등)로 그 URL이 실제 서비스 도메인이 맞는지 교차 확인한다.
- 여러 파일을 한 번에 작성했다면, 발행 직전에 그 안의 모든 외부링크를 모아 한 번 더 전수 검증한다.

### 5-2. 내부링크 — "더 읽어보기" 섹션 필수

같은 사이트의 다른 글로 연결하는 내부링크는 자유 서술 중간이 아니라, **글 맨 끝(FAQ 다음, 마무리 면책 문구 앞)에 아래 정확한 형식으로** 넣는다. `더 읽어보기`라는 문구가 정확히 `**더 읽어보기**`로 등장하는 지점부터 그 뒤의 링크들은 새 탭 처리에서 제외되어 같은 탭에서 자연스럽게 이동한다 — 그래서 이 마커를 정확히 지켜야 한다.

```markdown
**더 읽어보기**

- [관련 글 제목1](/blog/관련글슬러그1)
- [관련 글 제목2](/blog/관련글슬러그2)
```

- 관련 글이 실제로 사이트에 존재할 때만 링크를 건다. 아직 없는 글을 미리 링크로 걸지 않는다(깨진 링크 방지).
- 2~3개 정도가 적당하다. 같은 category이거나 주제가 이어지는 글을 우선 선택한다 (`src/utils/related-posts.js`의 점수 로직과 같은 원리).
- 이 사이트는 현재 글 수가 적어 관련 글이 없을 수 있다. 그런 경우 **"더 읽어보기" 섹션 자체를 생략**한다 — 빈 섹션이나 링크 없는 마커를 남기지 않는다.

### 5-3. 글 작성 시 링크 처리 절차 — 추천 → 컨펌 → 적용

새 글을 쓸 때 링크를 무작정 바로 박아넣지 않고, 아래 순서로 진행한다.

1. 본문을 훑어서 이미 발행된 다른 글과 겹치는 개념·제도가 나오면, "더 읽어보기"에 넣을 후보로 자연스럽게 삽입을 제안한다 — 판단 기준은 같은 `category`이거나 `eligibility`가 겹치는 글.
2. 정부 공식 출처가 필요한 대목(제도 근거, 정확한 수치)에는 웹 검색으로 신뢰할 수 있는 출처(정부 부처 공식 사이트, 관련 공공기관 공지) 1~3개를 후보로 찾아 추천한다.
3. **추천만 하고 바로 파일에 반영하지 않는다.** 사람의 컨펌을 받은 뒤에만 실제로 적용한다.
4. 신뢰 출처 우선순위는 9장(팩트체크 원칙)과 동일: 정부 부처·공공기관 공식 사이트 > 법령/고시 원문 > 신뢰할 수 있는 언론 보도. 블로그·포럼·SEO성 글은 출처로 쓰지 않는다.

### 5-4. 제목을 수정했다면 — 앵커 텍스트 재점검

이미 발행된 글의 제목을 나중에 고치면, **그 글을 "더 읽어보기"에서 링크하고 있는 다른 글들의 링크 표시 문구(앵커 텍스트)도 함께 점검**한다. 앵커 텍스트는 항상 대상 글의 **현재** 제목(또는 핵심부)과 일치해야 한다.

제목을 여러 개 한 번에 바꾸는 작업(예: SEO 개선을 위한 일괄 리라이팅) 이후에는, 수작업으로 하나씩 찾기보다 전체 md 파일에서 `title:`과 `[텍스트](/blog/slug)` 패턴을 전부 추출해서 slug별 실제 제목과 앵커 텍스트를 기계적으로 대조하는 전수 검사를 한 번 거친다 — "몇 개 수정했으니 그 글들만 확인"하는 방식은, 그 글들을 링크하는 *다른* 글들까지 놓치기 쉽다.

## 6. Frontmatter 스키마 (필수)

```yaml
---
title: "제목 (30자 내외, 핵심 변화가 드러나게)"
description: "요약 (검색결과 스니펫에 노출됨. 핵심 숫자·날짜 포함, 100자 내외)"
pubDate: "YYYY-MM-DD"
updatedDate: "YYYY-MM-DD"   # 생략 가능 — 발행 후 실제로 내용을 고쳤을 때만 추가
category: "self-employed" | "job-startup" | "youth" | "parenting" | "housing" | "senior" | "common"
contentTags: ["new"]         # 배열, 복수 선택 가능. 아래 6-3 참고
eligibility:                 # 생략 가능하지만 가급적 채울 것 (6-2 참고)
  ageRange: ["20s", "30s"]
  incomeLevel: "any"
  employmentStatus: ["any"]
tags: ["키워드1", "키워드2"]  # SEO용 자유 태그, 3~5개
thumbnail: "/images/글슬러그-01.png"  # 생략 가능. 소셜 공유용 대표 이미지
metric: "최대 250만원"       # 생략 가능. 카테고리 리스트 우측에 뜨는 핵심 수치 배지 (6-5 참고)
deadline: "2026-12-31"       # 생략 가능. 신청 마감일. 상시 모집이면 "ongoing" (6-6 참고)
---
```

### 6-1. `category` — 대상자 대분류 (하나만 선택)

| 값 | 의미 | 담는 내용 |
|---|---|---|
| `self-employed` | 소상공인·자영업 | 창업 초기 운영자금, 노란우산공제 |
| `job-startup` | 취업·창업 | 구직, 국민취업지원제도, 창업지원금 |
| `youth` | 청년 | 청년내일저축계좌 등 자산형성, 청년 전용 제도 |
| `parenting` | 육아·출산 | 한부모가족, 아동수당, 부모급여 |
| `housing` | 주거 | 전세자금대출, 주거급여, LH 임대주택 |
| `senior` | 중장년·시니어 | 국민연금, 은퇴 준비 |
| `common` | 전국민 공통 | 에너지바우처, 재난지원금 등 보편 지원 |

카테고리 선택이 애매하면(예: "청년 대상 주거 지원금") **더 구체적인 축**을 우선한다 — 이 예시라면 타겟이 명확한 쪽(`youth` 또는 `housing`). 판단이 안 서면 사람에게 물어본다.

### 6-2. `eligibility` — "나에게 맞는 혜택 찾기"/추천 기능용 메타데이터

정밀한 자격조건 판정이 아니라 **관련성 높은 글을 걸러내는 용도**다. 세부 자격조건은 본문에서 서술하고, 여기는 구간 단위로만 채운다.

- `ageRange`: 해당하는 나이대를 전부 선택 (`10s`~`60plus`). 나이 제한이 명확하지 않은 글(전 연령 대상)은 **생략** — 억지로 넓게 채우지 않는다.
- `incomeLevel`: `low`(기초생활수급 등 엄격한 소득 기준) / `middle`(중위소득 X% 이하 같은 중간 기준) / `any`(소득 무관). 본문에 소득 기준이 명시돼 있으면 반드시 채운다.
- `employmentStatus`: `employed`/`self-employed`/`unemployed`/`student`/`any` 중 해당하는 것 전부. 모호하면 `any`.

**확신이 없으면 생략한다.** 잘못 좁혀서 놓치는 것보다, 태깅을 비워 "전체 해당"으로 처리되는 게 안전하다.

### 6-3. `contentTags` — 콘텐츠 성격 (복수 선택 가능)

- `new`: 신설되거나 최근 변경된 제도 — `/new` 페이지에 자동 노출
- `regional`: 특정 지자체 한정 지원금 — `/regional` 페이지에 자동 노출
- `deep-guide`: 여러 제도를 종합한 심화 가이드

### 6-4. `tags` — SEO용 자유 태그

`category`/`contentTags`가 사이트 내부 분류·필터용이라면, `tags`는 검색엔진에 이 글의 핵심 키워드가 뭔지 알려주는 용도다.

- **개수**: 3~5개 정도로 제한한다. 남발하면 오히려 핵심 키워드가 희석된다.
- **내용**: 그 글에서 실제로 다루는 제도명·핵심 검색어를 그대로 쓴다. 예) `["양육비선지급제", "양육비선지급제신청방법", "양육비선지급제소득기준", "한부모가족"]`.
- **표기**: 지나치게 일반적인 단어(`정부지원금`, `혜택` 단독) 하나만 태그로 쓰지 않는다 — 검색 노출에 도움이 안 되고 사이트 전체 글에 다 걸릴 만큼 광범위해서 변별력이 없다. 그 글만의 구체적인 키워드 위주로 채운다.
- 제도명이 여러 표기로 불리는 경우(예: "청년내일저축계좌" vs "청년내일저축계좌 신청") 실제 검색될 법한 변형을 태그에 나눠 담아도 된다.

### 6-5. `metric` — 카테고리 리스트·검색결과 우측 수치 배지 (2026-07-20 신설, 같은 날 작성 규칙 보강)

카테고리 리스트 페이지와 `/find` 검색결과의 각 글 목록 항목 우측에 뜨는 짧은 강조 배지다("최대 250만원", "요건 폐지" 등). 목록을 훑을 때 숫자로 먼저 스캔할 수 있게 해주는 요소라, 이 사이트 특성(0장: 검색으로 들어와 조건만 빠르게 확인)과 잘 맞는다.

- **길이**: 8자 내외로 짧게. 본문 요약이 아니라 한눈에 들어오는 핵심 수치·상태 하나만 담는다.
- **생략 가능**: 값이 애매하거나 한 줄로 압축이 안 되는 글(여러 조건이 복합된 `deep-guide`형 글 등)은 생략한다. 생략하면 배지 자체가 안 뜨고 레이아웃도 자동으로 조정된다 — 억지로 채우지 않는다.

**값을 뽑는 순서(우선순위)**

글을 다 쓴 뒤, 아래 순서로 그 글의 "가장 먼저 눈에 들어와야 할 한 조각"을 고른다. 위에서부터 해당하는 첫 번째 유형을 쓰고, 그 유형 안에서 가장 핵심적인 값 하나만 남긴다.

1. **금액이 핵심이면 금액을 쓴다.** `[[SUMMARY]]` 블록에 금액·한도 라벨이 있다면 그 값을 그대로 가져온다.
   - 상한액이면 "최대"를 붙인다: `최대 250만원`, `최대 720만원`
   - 주기가 있으면 주기를 붙인다: `월 34만원`, `월 60만원`, `연 1800만원`, `일당 4.7만원`
   - 그 자체로 확정 금액이면 "최대" 없이: `25만원`, `519만원`
   - 상한과 하한을 둘 다 보여줘야 이해되는 글은 상한 기준 1일 단위처럼 가장 자주 언급될 단위로: `1일 6.8만원`
2. **금액이 없고 비율(%)이 핵심이면 비율을 쓴다.**
   - 할인율: `최대 20% 할인`
   - 금리·요율: `금리 4.5%`
3. **금액도 비율도 아니고, "무엇이 어떻게 바뀌었는가"가 핵심이면 상태변화형으로 쓴다.** 명사(바뀐 대상) + 변화동사(폐지/신설/완화/인상/확대 등) 조합으로 4~7자 안에 압축한다.
   - 요건·조건이 없어짐: `요건 폐지`, `혼인기간 요건 폐지`
   - 새 부담·항목이 생김: `자기부담금 신설`
   - 수치 기준이 오름: `보험료율 인상`
   - 절차·방식이 바뀜(금액으로 표현이 안 되는 구조 변경): `경매차익 선지급`

**금액형과 상태변화형이 둘 다 가능해 보이면 금액형을 우선한다** — 독자가 "얼마"를 "무엇이 바뀌었는지"보다 먼저 스캔하는 사이트 특성(0장) 때문이다. 다만 이 글의 핵심 주장 자체가 금액이 아니라 "제도 존재 여부/자격 요건"이라면(예: 소득기준 폐지로 대상이 넓어진 게 핵심이지 특정 금액이 핵심이 아닌 글) 상태변화형이 더 정확하다 — 어느 쪽이 이 글의 진짜 결론에 더 가까운지로 판단한다.

**참고 — 실제 21개 글의 값 (유형별 예시)**

| 유형 | 실제 값 |
|---|---|
| 금액형(최대) | 최대 250만원 · 최대 720만원 · 최대 360만원 · 최대 200만원 · 최대 2,255만원 · 최대 70만원 |
| 금액형(주기 명시) | 월 최대 13만원 · 월 34만원 · 일당 4.7만원 · 연 1800만원 · 1일 6.8만원 |
| 금액형(확정액) | 25만원 · 519만원 |
| 비율형 | 최대 20% 할인 · 금리 4.5% |
| 상태변화형 | 요건 폐지 · 혼인기간 요건 폐지 · 자기부담금 신설 · 보험료율 인상 · 경매차익 선지급 |

- **내용**: `[[SUMMARY]]` 블록의 최우선 항목과 같은 값을 쓰는 경우가 많다.

### 6-6. `deadline` — 신청 마감일 자동 표시 (2026-07-20 신설)

신청기간이 있는 글에 마감일을 명시해두면, 오늘 날짜가 그 날짜를 지났을 때 목록 페이지의 "마감" 배지와 글 상세 페이지의 안내 배너가 **자동으로** 뜬다(코드가 날짜만 비교하므로 별도 수동 작업 없이 계속 최신 상태로 유지된다).

- **특정 마감일이 있는 글**: `deadline: "2026-12-31"`처럼 정확한 날짜를 채운다.
- **상시/정기 모집이라 특정 마감일이 없는 글**: `deadline: "ongoing"`으로 명시한다. 단, 이 값은 예산·인원 캡 자체가 없어 사실상 상시인 법정 급여성 제도에만 쓴다.
- **예산이나 인원 캡이 있어 소진 시 조기 마감될 수 있는 글**: `deadline: "budget-limited"`로 명시한다(2026-07-21 신설). 접수 자체는 상시 열려 있지만 "선착순 O명", "예산 소진 시", "조기 마감 가능" 같은 표현이 본문·공식 출처에 있다면 이 값을 쓴다. `ongoing`과 묶으면 독자가 "천천히 신청해도 된다"고 오해할 수 있어 구분한다. 실시간 소진율은 대부분 공개되지 않으므로, 정확한 마감일을 알아내려 하지 않고 "소진 가능성이 있다"는 상태 자체를 표시하는 용도로 쓴다.
- **확신이 없으면 생략한다.** 9-3 원칙대로 상시/정기 여부 자체가 불확실한 상태에서 임의로 `"ongoing"`이나 아무 날짜나 채우면 안 된다 — 확인 전까지는 필드를 비워두고, 정부 1차 출처로 신청기간을 확인한 뒤에 채운다.
- **이 필드는 9-5(마감 정책은 글감에서 제외)와는 별개 축이다.** 9-5는 "글을 새로 쓸지 말지"를 정하는 주제 선정 단계 규칙이고, `deadline`은 "이미 쓴 글을 발행 후에도 자동으로 최신 상태로 보이게" 하는 표시 장치다. 새 글을 쓸 때 신청기간을 확인했다면, 그 시점에 바로 이 필드도 함께 채워둔다.
- 값을 채운 뒤에는 10장 체크리스트의 마감 관련 항목을 확인한다.

## 7. 파일 규칙

- 경로: `src/content/blog/영문-슬러그.md`
- 슬러그: 제도명을 영문 케밥케이스로 (예: `childcare-advance-payment-income-limit-abolished`). 파일명이 `_`나 `zzz` 같은 접두어로 시작하면 Astro가 콘텐츠 컬렉션에서 제외하니 피할 것.
- 인코딩: UTF-8, LF 개행
- 이미지 파일: `public/images/` 아래 저장, 본문에서는 `/images/파일명` 경로로 참조

### 7-1. 파일의 첫 줄은 반드시 frontmatter `---`여야 한다

Astro의 콘텐츠 컬렉션 파서는 **파일의 첫 줄**이 `---`일 때만 그 블록을 frontmatter로 인식한다. 발행 전 검수용 체크리스트를 HTML 주석(`<!-- ... -->`)으로 파일 맨 위에 남겨두면, frontmatter 앞에 다른 줄이 끼게 되어 `title`, `description`, `category` 등 모든 필드가 "Required"(없음) 에러로 빌드가 실패한다 — 개별 필드 값이 잘못된 게 아니라 frontmatter 블록 자체를 못 읽는 것이라 에러 메시지만 보면 원인이 헷갈리기 쉽다.

- **검수용 체크리스트가 필요하면 frontmatter `---` 뒤, 본문보다도 뒤 — 즉 파일 맨 끝에 HTML 주석으로 붙인다.** 발행 직전에 그 주석 블록만 통째로 지우면 된다.
- 파일을 새로 만들 때는 저장 직후 `head -1 파일명.md`로 첫 줄이 정확히 `---`인지 확인하는 습관을 들인다. 주석 블록 바로 뒤에 빈 줄이 남아도 같은 문제가 재발하니(빈 줄도 `---`가 아니므로), 첫 줄 자체가 `---`인지까지 확인한다.
- 여러 글을 한 번에 작성해 전달할 때는, 전달 직전에 모든 파일을 대상으로 첫 줄이 `---`인지 일괄 검사한다.

## 8. 새 카테고리 추가 시 체크리스트

지금 7개 카테고리(`self-employed`/`job-startup`/`youth`/`parenting`/`housing`/`senior`/`common`)를 늘리거나 이름을 바꿀 일이 생기면, 아래 **4곳을 전부** 고쳐야 빌드가 성공하고 실제로 메뉴에 노출된다. 카테고리 이름 자체는 `src/data/categories.js` 한 곳에 모아뒀지만(중복 정의 방지), 그것만으로는 부족하고 나머지 3곳도 함께 손봐야 한다 — 하나라도 빠뜨리면 빌드 에러가 나거나, 빌드는 되는데 메뉴에 안 보이는 식으로 조용히 누락된다.

1. **`src/content/config.ts`** — `category`의 `z.enum([...])`에 새 카테고리 값을 추가한다. **이걸 빠뜨리면 새 카테고리로 태깅한 글이 스키마 검증에서 빌드 에러로 죽는다.** (가장 놓치기 쉬운 곳)
2. **`src/data/categories.js`** — `CATEGORY_NAMES`(짧은 표기, 네비/카드용)와 `CATEGORY_NAMES_LONG`(풀어 쓴 표기, 카테고리 페이지 타이틀용) 둘 다에 새 항목을 추가한다.
3. **`src/pages/category/[cat].astro`** — `getStaticPaths()`의 경로 목록에 새 카테고리를 추가하고, `catDesc` 객체에 그 카테고리의 소개 문구(2문장)를 추가한다.
4. **`src/layouts/BaseLayout.astro`** — 헤더 `nav` 객체에 전체 라벨(`lbl-full`)과 모바일 짧은 라벨(`lbl-short`)을 추가하고, 실제 `<nav class="nav">` 마크업에도 링크를 추가한다.

**참고**: `src/components/PostCard.astro`, `src/layouts/PostLayout.astro`, `src/pages/find.astro`는 모두 `categories.js`를 가져다 쓰는 구조라 2번만 고치면 자동으로 반영된다 — 따로 손댈 필요 없다.

## 9. 팩트체크 · 출처 원칙

- 정부 지원금은 **숫자(금액, 기한, 소득기준)가 생명**이다. 확인 안 된 숫자를 추정해서 쓰지 않는다.
- 발표 시점과 시행 시점이 다른 경우가 많다 — 반드시 구분해서 서술한다("O월 O일부터 시행" vs "O월 O일 발표").
- 정보가 부처마다, 언론사마다 다르게 보도되는 경우 교차 검증한다. 숫자가 출처마다 갈리면 가장 공식적인 출처(정부 부처 보도자료, 관련 공공기관 공지)를 우선하고, 불확실하면 "정확한 기준은 공식 채널에서 확인하라"고 명시한다.
- 마무리 면책 문구는 **모든 글에 필수**로 넣는다:
  > *이 글은 YYYY년 M월 기준 발표된 정보를 바탕으로 작성했어요. 세부 신청 요건과 절차는 시행일에 가까워지며 조정될 수 있으니, 실제 신청 전에는 [관련 기관명] 공식 채널에서 최신 정보를 확인하시기 바랍니다.*

### 9-1. 1차 출처가 JS 렌더링이라 안 읽힐 때 — 우회해서라도 반드시 원문을 확인한다

복지로(bokjiro.go.kr)처럼 자바스크립트로 콘텐츠를 그리는 사이트는 페이지를 직접 열어도 내용이 안 보일 수 있다. 이때 "원문을 못 읽었으니 검증을 생략한다"로 넘어가지 않는다.

- 페이지 직접 열람이 막히면, 검색으로 그 페이지의 스니펫을 찾아 대체 확인한다. 검색 결과에 원문 텍스트 일부가 스니펫으로 노출되는 경우가 많다.
- 스니펫으로도 확인이 안 되면, 그 사실 자체를 사람에게 보고한다 — "복지로 원문을 확인 못 했고, 2차 출처(OO블로그)만 근거로 썼다"는 걸 숨기지 않는다.
- 원문 확인 없이 2차 출처(은행 블로그, 개인 블로그, SEO 최적화 글 등)만으로 제도의 핵심 성격(상시/한시, 신설/개편 여부)을 단정하지 않는다.

### 9-2. "핵심 주장"과 "부가 수치"는 검증 강도를 다르게 건다

한 글 안에서도 오류의 무게가 다르다. 아래 두 가지를 구분한다.

- **핵심 주장** — 제목·부제·도입부에 들어가는, 그 글이 존재하는 이유가 되는 주장. "제도가 상시화됐다", "신설됐다", "한도가 늘었다", "마감이 없어졌다" 같은 것. 이게 틀리면 글 전체가 오정보가 된다.
- **부가 수치** — 본문 중간의 구체적 금액, 날짜, 조건 등. 이게 틀리면 정정 대상이지만 글의 존재 이유 자체는 무너지지 않는다.

**핵심 주장은 반드시 정부 1차 출처(부처 보도자료, 공식 공고, 소관 기관 공지)로 확인한다.** 은행 블로그·개인 블로그·SEO 최적화 블로그는 참고는 하되, 핵심 주장의 근거로 삼지 않는다 — 이런 출처들이 서로의 오류를 베껴서 여러 곳에서 같은 말을 한다고 해도, 그게 1차 출처를 대체하지 못한다.

### 9-3. "상시/언제든/연중" 표현은 특히 의심한다

이런 표현은 2차 출처에서 부정확하게 쓰이는 빈도가 높다. 본문에 쓰기 전에는, 반드시 접수기간이 명시된 공식 공고(복지로/소관 부처/지자체)를 찾아 실제로 상시인지 정기모집인지 확인한다. 공식 공고에 특정 날짜 범위의 "신청기간"이 명시돼 있다면, 그건 상시가 아니라 정기모집이다.

### 9-4. "지금 신청하세요/갈아타세요"류 글은 그 행동의 기한부터 확인한다

특정 행동(전환, 갈아타기, 신청)을 유도하는 글은, 그 행동이 가능한 기간이 이미 지나지 않았는지를 **오늘 날짜 기준으로** 먼저 확인한다. "아직 안 늦었어요" 같은 표현을 쓰려면, 실제로 안 늦었는지 — 오늘 날짜와 마감/전환 기간을 직접 대조해서 — 확인한 뒤에 쓴다.

### 9-5. 신청기간이 이미 지난 정책은 글을 작성하지 않는다

주제 선정 단계에서, 정부 1차 출처로 확인한 신청기간(접수 마감일)이 **오늘 날짜보다 이전**이면 그 정책은 글감에서 제외한다. 이미 마감된 모집으로 글을 쓰면, "신청하세요"라는 행동 유도 자체가 독자에게 무의미하거나 오해를 줄 수 있다.

- 정기 모집(연 1회 등 특정 기간만 접수)인 정책은 반드시 **다음 회차 신청기간**이 언제인지까지 확인한 뒤에 글감으로 채택할지 판단한다. 다음 회차가 확정되지 않았다면 그 사실을 그대로 쓰고("다음 모집 일정은 미정"), 이미 지난 회차를 마치 신청 가능한 것처럼 쓰지 않는다.
- 상시 모집인지 정기 모집인지 자체가 불확실하면, 9-3 원칙에 따라 먼저 확인하고 나서 이 판단을 내린다.
- 이미 발행된 글도 시간이 지나 신청기간이 지나면 오정보가 된다. 발행 후 일정 기간이 지난 "신청 유도형" 글은 주기적으로 재확인 대상으로 삼는 게 안전하다(11장 참고).

## 10. 발행 전 최소 검수 체크리스트

사람이 최종 검수할 때 아래만 빠르게 확인한다 (전면 재검토가 아니라 최소 확인 목적):

- [ ] **파일 첫 줄이 정확히 `---`인가** (검수용 HTML 주석이 앞에 남아있으면 frontmatter 전체가 안 읽혀 빌드 실패 — 7-1 참고)
- [ ] frontmatter의 숫자(금액, 날짜, 소득기준 %)가 본문과 정확히 일치하는가
- [ ] `category`가 실제 내용과 맞는가
- [ ] `eligibility`를 억지로 채우지 않았는가 (확신 없으면 비워뒀는가)
- [ ] `==하이라이트==`가 과하지 않은가 (2~3개 수준, 실제로 세어봤는가)
- [ ] **본문 개념어형 `<strong class="key">`가 1~2개 수준인가, 라벨형(`[[SUMMARY]]` 안)과 헷갈리는 자리에 쓰지 않았는가** (3-1 참고)
- [ ] **대상/금액/조건 섹션에 `[[SUMMARY]]` 요약 박스를 썼는가, 실제로 카드 형태로 렌더링되는가** (미리보기에서 대괄호 마커 텍스트가 그대로 보이면 항목 형식이 깨진 것 — 3-3 참고)
- [ ] **FAQ의 Q 줄 끝에 백슬래시(`\`)가 빠짐없이 붙어 있는가** (없으면 Q-A가 줄바꿈 없이 한 줄로 붙어 보임 — 2-4 참고)
- [ ] **외부링크가 최소 1개 이상 있는가** (공식 출처로 연결되는가) — **그 URL을 실제로 열어서 정상 작동하는지, 가장 정확한 신청처/근거 페이지가 맞는지 확인했는가** (5-1-1 참고, 도메인만 맞고 더 정확한 전용 페이지가 따로 있을 수 있음)
- [ ] **이미지를 썼다면**, 표로 대체 불가능한 정보였는지 근거가 있는가 / alt 텍스트와 캡션이 둘 다 있는가 / 캡션에 그림 번호가 붙어 있는가
- [ ] **캡처 이미지를 썼다면, 이용조건(공공누리 유형 등)을 사람이 직접 확인했는가** (Claude가 보고한 건 사실 확인까지이고, 최종 이용 가능 여부 판단은 사람 몫)
- [ ] "더 읽어보기" 섹션을 넣었다면 링크가 실제 존재하는 글인가 (깨진 링크 없는가), 그 대상 글의 링크 표시 문구(앵커 텍스트)가 대상 글의 현재 제목과 일치하는가
- [ ] **신청 방법 각 단계에 소요 기간이 확인됐다면, 그 단계 문장 안에 괄호로 병기했는가** (2장 8번 참고 — 절차와 기간이 따로 떨어져 있지 않은가)
- [ ] **범위 표현에 물결표(`~`)를 쓰지 않았는가** (`3~6개월`처럼 쓰면 취소선으로 깨짐 — `3-6개월`로 교정)
- [ ] **굵게(`**`)가 괄호 바로 뒤에서 조사와 붙어 깨지지 않았는가** (`**포인트(CEU)**를`처럼 쓰면 강조가 풀림)
- [ ] 마무리 면책 문구가 있는가
- [ ] 시행일이 아직 안 지났다면 "예정" 표현이 맞게 쓰였는가 (이미 시행됐는데 "예정"으로 써놓는 실수 방지)
- [ ] **이 글의 핵심 주장을 정부 1차 출처로 직접 확인했는가** (9-1, 9-2 참고 — 은행/개인 블로그만 근거로 쓰지 않았는가)
- [ ] **"상시/언제든/연중" 표현을 썼다면, 공식 공고에 실제 신청기간이 없는 게 맞는지 재확인했는가** (9-3)
- [ ] **오늘 날짜 기준으로 신청기간·전환기간이 지나지 않았는가** (9-4, 9-5)
- [ ] **`metric`을 채웠다면 8자 내외로 짧은가, 본문·SUMMARY의 핵심 수치와 일치하는가** (6-5 참고)
- [ ] **`deadline`을 채웠다면 정부 1차 출처로 확인한 값인가** — 상시/정기 모집이 불확실한 상태에서 임의로 채우지 않았는가, 특정 마감일이 있다면 정확한 날짜인가, 상시라면 `"ongoing"`으로 명시했는가 (6-6 참고)

## 12. 슬러그(URL) 변경 시 리다이렉트 규칙

이미 발행해서 검색엔진에 인덱싱된 글의 슬러그(파일명)를 나중에 바꾸면, 기존 URL로 들어오던 방문자와 검색엔진이 전부 404를 만난다. 슬러그를 바꿔야 한다면 `public/_redirects` 파일에 이전 경로 → 새 경로 리다이렉트를 반드시 추가한다.

```
/이전-슬러그/   /blog/새-슬러그   301
```

**지금은 이 파일이 아직 없다** — 실제로 배포하고 나서 슬러그를 바꿀 일이 생기면 `public/_redirects` 파일을 새로 만들고 위 형식으로 한 줄씩 추가하면 된다. 처음 만들 때는 맨 위에 위 형식을 참고 주석으로 남겨두면 다음에 헷갈리지 않는다.

- 한 번 발행한 글의 슬러그는 가급적 바꾸지 않는 게 최선이다. 이 규칙은 어쩔 수 없이 바꿔야 할 때의 안전장치다.
- 슬러그를 바꾼 글이 다른 글의 "더 읽어보기"에 링크되어 있다면, 그 링크의 `href` 경로도 새 슬러그로 함께 갱신한다(리다이렉트가 있어도, 애초에 최신 경로로 링크하는 게 더 깔끔하다).

## 13. 작업 세션 운영 방식

토큰 절약과 맥락 관리를 위해, 작업 종류별로 채팅방(대화)을 분리하는 게 효율적이다.

- **코드/기능 수정** (사이트 구조, 카테고리 추가, 디자인 변경 등) → 별도 채팅방
- **글 작성/발행** → 별도 채팅방
- **카피·문구 검토** → 별도 채팅방

같은 프로젝트 안에서 새 채팅방을 열어도, 이 문서가 프로젝트 지식(파일)으로 등록되어 있으면 규칙은 그대로 공유된다. 매번 규칙을 다시 설명할 필요 없이, 새 채팅방을 열고 바로 작업을 시작하면 된다.

### 13-1. 코드 세션 대기 중인 요청사항

글 작성 세션에서 발견했지만 md 파일 수정만으로는 해결이 안 되고 코드(CSS, remark 플러그인 등) 변경이 필요한 항목을 여기 적어둔다. 코드 세션에서 처리되면 이 목록에서 지운다.

**지금은 대기 중인 항목이 없다.** 아래 두 건은 2026-07-21 코드 세션에서 처리 완료되어 목록에서 지웠다.

- ~~FAQ A 답변 두 번째 줄 들여쓰기~~ → `remark` 플러그인에서 "A." 라벨을 별도 span으로 분리하고, `grid-template-columns: max-content 1fr`로 라벨 폭을 자동 계산하는 CSS Grid 방식으로 처리. em 단위 폭 추정 방식은 실제 폰트 렌더링과 어긋나 폐기(16-2 참고).
- ~~`deadline: "budget-limited"` 프론트엔드 표시~~ → `config.ts` 스키마에 `'budget-limited'` 리터럴 추가, 목록 카드·글 상세 배너에 "예산 소진 시 조기 마감될 수 있어요" 안내를 상시·고정으로 노출하는 앰버 톤 배지/배너 신설(`ongoing`과 시각적으로 구분). 날짜 비교 로직에는 `"ongoing"`/`"budget-limited"`를 먼저 걸러내는 방어 코드 적용(16-6 참고).

## 14. 작업 원칙 — 규칙이 실제로 지켜졌는지 확인하는 방법

앞의 1~13장이 "무엇을 써야 하는가"에 대한 콘텐츠 규칙이라면, 이 장은 "규칙이 실제로 지켜졌는지 어떻게 확인하고 고치는가"에 대한 작업 방식 원칙이다.

### 14-1. "가이드에 규칙이 있다"와 "실제로 반영됐다"는 별개다

규칙 문서를 만드는 것과, 그 규칙이 실제 산출물에 지켜졌는지 확인하는 것은 서로 다른 작업이다.

- 여러 항목을 한 번에 점검할 때는(예: "이전에 요청한 9가지가 다 반영됐나 확인해줘") 각 항목을 **실제 코드/콘텐츠에서 직접 grep하거나 값을 읽어서** 확인한다. "아마 반영했을 것"이라는 추정으로 답하지 않는다.
- 특히 체크리스트성 규칙(링크 개수, 하이라이트 개수, 이미지 alt 유무)은 규칙을 세운 시점과 실제로 그 규칙을 지켜 글을 쓴 시점이 다른 대화(세션)일 수 있다는 걸 감안한다. 규칙만 있고 적용은 안 됐을 가능성을 항상 열어둔다.

### 14-2. 버그·동작 방식은 추측하지 말고 실제로 재현해서 검증한다

다른 사이트나 예전 대화에서 가져온 "이렇게 동작한다"는 정보는, 지금 이 사이트의 실제 코드에서 같은 조건으로 재현되는지 확인하기 전까지는 미확정 정보로 취급한다. 특히 마크다운 렌더링, CSS 스타일 적용처럼 코드 버전이나 설정에 따라 달라질 수 있는 부분은 "예전에 그랬다"가 아니라 "지금 이 코드에서 실제로 그런지" 기준으로 검증한다.

같은 원칙이 **사실관계(정책 내용) 검증에도 적용된다** — 2차 출처(다른 블로그)가 "이렇게 바뀌었다"고 서술해도, 그게 지금 이 시점 정부 1차 출처에서 실제로 확인되는지 검증하기 전까지는 미확정 정보로 취급한다(9-1~9-5 참고).

### 14-3. 이미지·사이즈 규격은 "표시 크기"와 "저장 해상도"를 구분해서 문서화한다

사이즈 관련 규칙을 쓸 때는 항상 "이게 표시 크기 기준인지 저장 해상도 기준인지"를 명시한다(4-3 참고). 이 사이트의 본문 컨테이너 폭은 720px이므로, 저장 규격(1200px)은 레티나 디스플레이 대응용 여유분이지 화면 표시 크기가 아니다.

### 14-4. UI 요소는 "어떻게 처리할지"를 명시적으로 지시하지 않으면 이전 상태가 관성적으로 유지된다

이미지·레이아웃 관련 지시를 받으면 아래 세 가지를 항상 구체적으로 확인하거나, 확인이 어려우면 가정한 내용을 명시적으로 밝히고 진행한다.

- 크롭 여부("전체 이미지" vs "특정 부분만")
- 정렬 기준(가로/세로 중앙, 좌측/우측 정렬 등)
- 배경 마스크(원형, 사각형, 둥근 사각형 등)

### 14-5. 강조 요소는 개수 상한이 있어도 인플레이션이 일어난다

강조는 쓰다 보면 "이것도 중요한데" 싶어서 계속 늘어나는 경향이 있어, 상한선까지 자연스럽게 수렴한다. 규칙을 상한선(예: "최대 4개")으로 정하면 실제로는 그 상한까지 채워지는 경향이 있다. 정말 필요한 개수보다 넉넉하게 상한을 잡지 않는다(3장 하이라이트 규칙에 이미 반영됨).

같은 현상이 하이라이트뿐 아니라 개수 범위로 정한 다른 요소에도 나타난다. 2026-07-16 기준 발행 대기 21개 글을 전수 확인한 결과, FAQ는 "2~4개" 범위인데도 21개 전부 정확히 3개로 수렴해 있었다(9장 참고 — 규칙과 실제 반영은 별개이므로, "범위 규칙"을 세울 때는 실제로 몇 개가 쓰이고 있는지 주기적으로 세어보는 게 안전하다).

### 14-6. 브랜드명·고유명사 표기는 "이름 자체로 노출되는 자리"와 "문장 속에서 서술되는 자리"를 기준으로 나눈다

"혜택PICK!!"과 "혜택PICK" 두 표기를 병행할 때, 파일별로 정하지 않고 **역할 기준**을 세운다:

- **이름 자체로 노출**: 로고, 페이지 `<title>` 태그, 저작권 표시(©), 구조화 데이터(JSON-LD)의 `name` 필드 → 정식 명칭 그대로("혜택PICK!!")
- **문장 속 서술**: `description`, 본문, alt 텍스트, 법조항 서술문 → 축약형("혜택PICK")

표기가 여러 형태로 병행되는 고유명사(제도명, 브랜드명 등)가 생기면 이 방식으로 "왜 이 자리는 이 표기인가"의 기준을 먼저 세운다.

### 14-7. 사실관계 재발방지 사건 요약 (2026-07-14)

22개 글을 일괄 발행한 뒤 이 중 최소 1개(청년월세지원)에서 핵심 주장 자체가 틀린 것으로 확인됐다 — "2026년부터 상시 신청으로 바뀌었다"고 썼지만, 정부 1차 출처(복지로·지자체 공고)를 확인한 결과 실제로는 매년 정해진 기간에만 접수하는 정기모집이었다. 근거는 2차 출처(은행 블로그)였고, 그 2차 출처들이 서로 같은 오류를 반복하고 있었을 뿐 1차 출처로 확인된 적이 없었다.

## 16. 코드/디자인 작업 원칙 (2026-07-20 신설)

앞 장들이 "무엇을 써야 하는가"(콘텐츠)와 "규칙이 지켜졌는지 어떻게 확인하는가"(14장, 작업 방식 일반론)라면, 이 장은 **코드 세션(리브랜딩, 기능 추가, UI 수정)에서 반복적으로 나타난 실패 패턴과 그 대응 원칙**이다. 2026-07-20 네이비 리브랜딩 세션에서 얻은 인사이트를 정리했다.

### 16-1. 미리보기(시안)를 실제 코드에 옮길 때는 항목 단위로 대조한다

디자인 시안을 승인받은 뒤 실제 코드에 반영하는 작업에서, "이 섹션은 이미 구현했다"는 인상만으로 다음 섹션으로 넘어가면 시안의 일부 요소가 통째로 누락되기 쉽다. 실제로 이 세션에서 "카테고리별로 찾아보기" 섹션 전체, 통계 카드, breadcrumb, 읽는 시간 표시, 카드형 그리드(리스트형으로 잘못 유지됨) 등이 여러 차례 걸쳐 누락됐다가 사용자가 스크린샷으로 지적한 뒤에야 발견됐다.

- 시안 HTML과 실제 렌더링 결과를 **섹션이 아니라 개별 요소(문구, 배지, 버튼, 카드 개수 등) 단위로** 하나씩 grep하거나 대조한다.
- "섹션 A는 확인했으니 섹션 B, C도 비슷한 수준일 것"이라고 넘겨짚지 않는다. 섹션마다 별도로 검증한다.
- 사용자가 "왜 이렇게 나오지?" 또는 스크린샷으로 문제를 지적하면, 그 지점만 고치고 끝내지 말고 **같은 종류의 누락이 다른 페이지/섹션에도 있는지 전체를 재검증**한다(실제로 이 패턴이 이 세션에서 3회 반복됐다).

### 16-2. 텍스트 폭을 CSS 숫자값으로 추정하지 않는다

"A." 같은 짧은 라벨 뒤에 텍스트를 정렬할 때, `text-indent`/`padding-left`에 `em` 단위 추정치를 쓰는 방식은 가변폭 웹폰트(Pretendard 등)에서 실제 렌더링 폭과 어긋난다. 이 세션에서 `1.9em` → `1.35em`으로 두 차례 값을 조정해도 계속 안 맞았다.

- **폭을 추정해야 하는 상황이면, CSS 자체가 폭을 계산하게 만든다.** `grid-template-columns: max-content 1fr` 같은 방식을 쓰면 브라우저가 실제 콘텐츠 크기에 맞춰 첫 컬럼 폭을 자동 계산하므로, 폰트가 무엇이든 항상 정확하다.
- 이 원칙은 hanging indent뿐 아니라 "라벨+내용" 형태의 모든 정렬(FAQ, 범례, 키-값 목록 등)에 적용된다.

### 16-3. `display: flex` 안의 텍스트는 별도 요소로 감싼다

`display: flex`인 리스트 항목에 `::before`(숫자·아이콘 등 가상요소)와 텍스트를 나란히 두면, 텍스트가 "익명 flex item"으로 취급되어 `min-width` 같은 속성을 줄 수 없다. 이 상태에서 링크가 섞인 긴 텍스트가 줄바꿈되면 단어나 링크 중간이 이상한 지점에서 잘린다.

- flex 컨테이너 안의 텍스트 콘텐츠는 항상 실제 요소(`<span>` 등)로 감싼다. 마크다운이 생성하는 콘텐츠라면 rehype 플러그인으로 자동 감싸는 방법을 쓴다(`rehypeOlContent` 참고 — `markdown.mjs`).
- 감싼 요소에 `min-width: 0`(또는 `overflow-wrap: break-word`)을 명시해야 정상적으로 줄바꿈된다.

### 16-4. 전역 CSS 셀렉터는 같은 태그를 쓰는 특수 컴포넌트를 `:not()`으로 배제한다

`.prose ol`처럼 특정 부모 아래 모든 `<ol>`에 스타일을 거는 규칙을 만들 때, 그 부모 안에 태그는 같지만 용도가 다른 특수 컴포넌트(예: 목차의 `<ol class="toc-list">`)가 있으면 두 스타일이 충돌한다.

- 이 버그는 **두 컴포넌트가 같은 부모 안에 실제로 공존하는 상황에서만 드러난다.** 이 세션에서는 목차가 데스크톱에서는 사이드바로 빠져나가 `.prose` 밖에 있어서 문제가 안 보였는데, 모바일에서는 `.prose` 안에 그대로 남아 있어서 원형 배지 스타일과 충돌해 숫자가 겹쳐 보였다.
- 전역 스타일을 새로 만들 때는 "이 태그를 쓰는 다른 특수 컴포넌트가 이 부모 안에 있을 수 있는가"를 확인하고, 있다면 `:not(.특수클래스)`로 명시적으로 배제한다.
- 이런 버그는 데스크톱 화면만 확인해서는 못 잡는다. 반응형 분기점(모바일/데스크톱)마다 별도로 검증한다.

### 16-5. 마크다운이 생성하는 HTML은 컴포넌트 스코프 CSS의 적용 범위를 다시 확인한다

Astro 같은 프레임워크의 컴포넌트별 스코프 CSS(`data-astro-cid-*`)는 그 컴포넌트가 직접 렌더링하는 엘리먼트에만 자동으로 붙는다. 마크다운 파이프라인(remark/rehype)이 생성하는 순수 HTML은 이 스코프가 안 붙는다.

- 마크다운발 콘텐츠에 스타일을 걸 때, 그 스타일이 정의된 파일(`BaseLayout.astro` 등)의 스코프에 실제로 걸리는지 확신하지 말고 빌드 결과 HTML에서 확인한다.
- 이 콘텐츠를 다른 컴포넌트 소유의 엘리먼트(예: JS로 사이드바에 옮겨진 목차) 안으로 이동시키는 경우, 그 다른 컴포넌트의 스코프 CSS로 오버라이드하려면 일반적인 조합 셀렉터로는 안 걸릴 수 있다 — `is:global`을 명시해서 스코프 자체를 벗어나야 한다.

### 16-6. 시간 기반 자동 표시 기능은 날짜 경계값을 실제 계산으로 검증한다

"발행 후 N일간 NEW 배지", "마감일이 지나면 마감 배지" 같은 시간 기반 자동 판정 로직은, 경계값(오늘이 정확히 N일째인 경우, 마감 당일인 경우)에서 의도와 다르게 동작하기 쉽다. 이 세션에서 NEW 배지 로직이 `diffDays <= 7`이어야 할 걸 `diffDays < 7`로 잘못 짜서, 7일째 되는 글이 하루 일찍 NEW 배지를 잃는 버그가 있었다.

- "N일간 유지"라는 요구사항을 코드로 옮길 때, 경계를 포함하는지(`<=`) 배제하는지(`<`)를 말로 먼저 명확히 정의한 뒤 그대로 구현한다.
- 시:분까지 비교하면 발행/마감 시각에 따라 결과가 흔들리므로, 날짜(자정 기준)로 정규화한 뒤 정수 일수로 비교한다.
- 배포 전에 실제 오늘 날짜와 몇 가지 테스트 값(경계 정확히 걸치는 값 포함)으로 계산 결과를 검증한다. 이 세션에서는 실제로 과거 날짜를 임시로 넣어 빌드해서 배지가 올바르게 뜨고 사라지는지 확인한 뒤 원상복구하는 방식을 썼다.

### 16-7. 빌드 성공은 검증의 시작이지 끝이 아니다

`astro build`가 에러 없이 끝났다는 것은 "코드가 문법적으로 유효하다"는 것만 보장하지, "요청한 대로 렌더링됐다"는 것은 보장하지 않는다.

- 빌드 후에는 실제로 생성된 HTML에서 목표한 값(텍스트, 클래스, 속성, 개수)이 정확히 들어갔는지 grep이나 파싱으로 직접 확인한다.
- 여러 글/페이지에 걸친 변경이면, 1개만 확인하고 끝내지 말고 전체 개수(예: "21개 글 중 21개 모두 반영됐는가")를 세어 검증한다.
- CSS 변경은 값이 선언부에 있는지("정의됐다")와 실제 엘리먼트에 그 클래스·속성이 붙어 있는지("적용됐다")를 모두 확인해야 한다. 정의만 있고 마크업이 안 따라오면 스타일이 안 먹는다.
